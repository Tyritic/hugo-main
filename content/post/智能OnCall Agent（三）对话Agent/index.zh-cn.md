## 💬 对话Agent简介
你可以把它看作是一个能够像真人一样理解问题、调用知识库并给出精准回答的中台助手。它最重要的使命就是帮助团队挡掉高频的重复咨询，加速问题解决，从而提高整体的工作效率。
简单来说，对话Agent核心就是AI+知识库的模式。它把团队中散落的文档、经验和历史数据，转
化成可随时调用的活知识，让你的经验和知识不再因为人员流动而流失。

### 🎯 核心定位

对话Agent的核心定位是**团队知识的一站式查询入口**。无论是值班人员想了解某个错误码的含义，还是运维新人想查询服务接入文档，只需用自然语言提问，即可获得准确、专业的回答。主要服务场景
- 业务方支持
    - 传统模式:业务方->中台同学(反复提问)->手动翻文档->回复。
    - Agent模式:业务方->对话Agent(自动回复)->解决不了再找中台。 
    - 效果:自动解答接入文档、接口规范等问题，将中台同学从人工客服中解放出来。
- 值班自救
    - 值班人员收到告警->凌晨迷糊查手册->尝试修复->可能延误。
    - Agent模式:值班人员将告警复制给Agent->Agent自动检索《故障处理手册》并推送解决方案->人工快速确认。
    - 效果:在不到10秒内返回修复步骤，大大压缩了值班处理告警的时间，降低了响应延误风险。
- 工单预处理
    - 传统模式:新工单->研发逐条查看->简单问题占用时间。
    - Agent模式:新工单->Agent检索历史案例库->简单问题自动回复/自动总结->复杂问题再流转给人工。
    - 效果:提前过滤掉简单问题，让团队能将精力聚焦在真正需要人工解决的疑难杂症上。

### 🔗 与知识库的关系

对话Agent与知识库Agent形成**底层支撑与上层应用**的关系：

- **知识库**：负责文档的加载、分片、向量化和索引构建，是整个系统的数据底座
- **对话Agent（本文）**：负责接收用户Query、调度检索、构建上下文、调用LLM生成回答，是面向用户的交互窗口

对话Agent依赖知识库Agent提供的检索能力：当用户提问时，对话Agent会将问题转化为向量，在知识库中检索相关文档片段，然后结合检索结果生成回答。

### ⚡ 在线检索流程

对话Agent的在线检索流程包含以下关键步骤：

#### Multi-Query 策略

为了提高召回率，对话Agent采用Multi-Query策略：从多个角度生成查询变体，合并结果以获得更全面的检索结果。

```go
mqr, err := multiquery.NewRetriever(ctx, &multiquery.Config{
    RewriteLLM: llm,
    Reranker:   reranker,
    Retrievers: []interface{}{
        vectorRetriever,
        bm25Retriever,
    },
    TopK: 10,
})
```

#### 混合检索与多路召回

系统同时执行两种检索策略：

- **向量检索**：通过语义相似度匹配，找到与用户问题含义相近的文档片段
- **BM25检索**：通过关键词匹配，找到包含用户问题中关键术语的文档片段

两种检索结果经过融合排序，生成粗排候选集。

#### ReRank 重排序

在精排阶段，使用bge-reranker模型对候选文档进行相关性重排序，筛选出最匹配问题上下文的Top-K结果，交给LLM生成最终回答。

### 📝 对话上下文构建

检索结果需要经过精心组织的上下文构建，才能有效地注入Prompt：

- **文档片段**：最相关的正文内容
- **来源信息**：文档标题、章节路径、页码等，便于用户溯源
- **相关度分数**：帮助判断结果的可信度

```go
func BuildContext(docs []schema检索结果) string {
    var sb strings.Builder
    sb.WriteString("【参考文档】\n\n")
    for i, doc := range docs {
        sb.WriteString(fmt.Sprintf("【文档 %d】\n", i+1))
        sb.WriteString(fmt.Sprintf("来源：%s\n", doc.Source))
        sb.WriteString(fmt.Sprintf("内容：%s\n\n", doc.Content))
    }
    return sb.String()
}
```

## 🧪 实际对话示例

### 错误码查询

**用户提问**：错误码120000001是什么原因？

**Agent回答**：
> 错误码120000001表示**下游服务调用超时**。
>
> 根据《XX系统错误码手册》，该错误码的含义是：在调用下游服务时，等待响应超时。可能的原因包括：
> - 下游服务负载过高，响应延迟
> - 网络抖动导致连接中断
> - 下游服务存在死锁或慢查询
>
> **建议处理步骤**：
> 1. 检查下游服务的健康状态和监控面板
> 2. 查看最近是否有发布变更
> 3. 如果超时比例持续上升，建议联系下游服务负责人确认
>
> 📖 来源：《XX系统错误码手册》- 第3章 服务超时类错误

---

### 服务接入咨询

**用户提问**：我想接入用户服务，需要准备什么？

**Agent回答**：
> 接入用户服务需要准备以下内容：
>
> **1. 基础信息**
> - 服务名称和负责人联系方式
> - 服务的接口文档（OpenAPI 3.0格式）
>
> **2. 鉴权配置**
> - AppKey 和 AppSecret（向平台申请）
> - 回调地址（用于接收异步通知）
>
> **3. 测试环境**
> - 测试环境的部署地址
> - 测试账号和权限配置
>
> **4. 监控告警**
> - 告警接收人列表
> - 告警阈值配置
>
> 📖 来源：《服务接入指南》- 第2章 接入准备

---

### 故障处理询问

**用户提问**：数据库连接池满了怎么处理？

**Agent回答**：
> 数据库连接池满是一个比较严重的问题，可能导致服务不可用。以下是推荐的处理步骤：
>
> **紧急处理**：
> 1. **立即告警**：通知相关开发和运维人员
> 2. **查看监控**：确认连接池的使用情况和变化趋势
> 3. **临时扩容**：如果配置允许，临时增大连接池大小
>
> **根因分析**：
> 1. 检查是否有慢查询占用连接
> 2. 查看连接泄漏（未正确关闭连接）
> 3. 分析业务流量是否异常增长
>
> **预防措施**：
> 1. 配置连接池的合理大小和超时时间
> 2. 添加连接池监控告警
> 3. 定期巡检数据库性能
>
> 📖 来源：《故障处理手册》- 数据库类问题

## ✅ 总结与后续

### 当前能力

对话Agent目前已实现以下能力：

- **自然语言问答**：支持用户用自然语言查询知识库
- **多角度召回**：通过Multi-Query策略，提高检索召回率
- **混合检索**：结合向量检索和BM25，兼顾语义和关键词匹配
- **来源追溯**：每条回答都附带信息来源，方便用户核实

### 局限性

当前实现仍存在一些局限：

- **上下文长度限制**：受限于LLM的上下文窗口，无法处理过长的问题
- **多轮对话能力**：目前主要支持单轮问答，多轮对话仍在探索中
- **实时性要求**：知识库更新后，需要重新索引才能被检索到

### 后续优化方向

- **多轮对话支持**：实现上下文记忆，支持追问和澄清
- **知识库增量更新**：支持知识库的实时增量更新，无需全量重建
- **对话能力增强**：支持更多类型的查询，如对比分析、趋势分析等
